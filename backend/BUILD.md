# Freezing the backend (PyInstaller)

The Tauri shell spawns the Python backend as a subprocess over stdio (`docs/ipc_protocol.md`).
In dev mode that subprocess is `backend/.venv/Scripts/python.exe backend/run.py`. For a
shippable installer there is no guarantee a matching Python interpreter exists on the
target machine, so the backend is frozen into a standalone executable with PyInstaller
and wired into Tauri as an `externalBin` sidecar.

## Building the frozen executable

```
cd backend
.venv/Scripts/python.exe -m pip install "pyinstaller==6.21.0" "pyinstaller-hooks-contrib==2026.6"
.venv/Scripts/python.exe -m PyInstaller --onefile --name eiclean-backend --console run.py
```

This produces `backend/dist/eiclean-backend.exe` (~44MB) plus `backend/eiclean-backend.spec`
and `backend/build/`. The spec file is the PyInstaller-generated default (no custom
`hiddenimports`/`datas` were needed — `pyinstaller-hooks-contrib` covers `pymobiledevice3`'s
dependencies out of the box) so it, `build/`, and `dist/` are all gitignored and
regenerated by the command above rather than committed.

`--console` is intentional: the sidecar communicates over stdin/stdout, so it must keep a
real console subsystem rather than building as `--windowed`.

## Validated behavior

The frozen exe was tested directly (not just assumed to work) by feeding it newline-delimited
JSON-RPC requests and reading the responses:

```
cmd /c "dist\eiclean-backend.exe < msg.txt"
```

`cmd.exe` file redirection was used deliberately — piping through Git-Bash/MSYS produced
empty stdout despite the process running correctly (a shell/pipe compatibility quirk, not a
backend bug), and .NET's `Process.StandardInput` prepends a UTF-8 BOM that the backend's
`json.loads` chokes on. Neither of those match how Tauri's `tokio::process` actually talks to
the child, so neither was a reliable test harness; a plain `subprocess.Popen` with stdin kept
open (mirroring `tokio::process::Command`) reproduces real behavior correctly.

Confirmed against the frozen exe:
- `device.list` and `settings.get` return correct JSON-RPC responses, matching dev-mode
  (unfrozen) behavior.
- DB init, logging, the device-watcher loop, and USB selective-suspend mitigation all start
  correctly on launch.
- The full `cargo tauri build` release pipeline (see below) launches `app.exe`, which spawns
  the frozen sidecar successfully and the sidecar's device-watcher loop runs continuously.

**Known gap (as of this frozen-exe test)**: this dev machine had no Apple Mobile Device
Support / usbmuxd installed at all (confirmed via the exact exception surfaced in
`%APPDATA%\EiClean\logs\backend.log`: `pymobiledevice3.exceptions.ConnectionFailedToUsbmuxdError`
/ `WinError 1225 connection refused`), so `device.connect`'s success path against a real,
trusted device had not been exercised end-to-end here. `device.list`/`settings.get` were
confirmed identical between frozen and dev-mode builds under this same missing-driver
condition, which is evidence (not proof) that freezing itself isn't the cause of any
`device.connect` slowness — see the fix below.

The backend now detects this condition itself (`driver_missing`/`driver_available` events,
`backend/app/device/discovery.py`) instead of silently hanging on "Connect your iPhone" — see
docs/DEVELOPMENT.md's "Known gaps". After installing the driver via the UI's prompt, a real
iPhone did connect and pass the trust handshake in dev mode for the first time (see
docs/DEVELOPMENT.md for the `AfcService` async bug that surfaced immediately after) — but this
has not yet been re-verified against the **frozen** exe specifically, only dev mode.

### Bug found via this testing: unbounded pairing wait

`wait_for_trust()` (`backend/app/device/pairing.py`) previously had no timeout on its initial
`is_paired()` call, so a slow/hanging usbmux round-trip (e.g. from a missing driver) could
block indefinitely with no IPC response. Fixed by bounding every individual `is_paired`/
`request_pairing` attempt with `settings.PAIRING_ATTEMPT_TIMEOUT_SECONDS` (default 10s),
independent from the overall user-facing `TRUST_PROMPT_TIMEOUT_SECONDS` (60s) budget for
"waiting on the user to tap Trust". Covered by the existing pairing test suite; no regression.

## Wiring into Tauri as a sidecar

1. Copy the frozen exe into `src-tauri/binaries/`, named per Tauri's `externalBin`
   convention (target-triple-suffixed):
   ```
   cp backend/dist/eiclean-backend.exe src-tauri/binaries/eiclean-backend-x86_64-pc-windows-msvc.exe
   ```
   `src-tauri/binaries/` is gitignored — it's a build artifact regenerated from
   `backend/dist/`, not a source file.
2. `src-tauri/tauri.conf.json` declares it under `bundle.externalBin`:
   ```json
   "externalBin": ["binaries/eiclean-backend"]
   ```
3. `src-tauri/src/sidecar.rs` resolves the path at runtime: dev builds
   (`cfg!(debug_assertions)`) use the venv `python.exe run.py` invocation as before; release
   builds resolve `eiclean-backend.exe` next to `app.exe` via `std::env::current_exe()`.

### Empirically confirmed: the target-triple suffix is stripped

Tauri's docs don't clearly state what happens to the `-<target_triple>` suffix on Windows
after bundling, so this was confirmed directly by running a full `cargo tauri build` and
inspecting the output rather than guessing:

- `target/release/eiclean-backend.exe` — suffix stripped, sitting next to `app.exe`.
- The generated NSIS script (`target/release/nsis/x64/installer.nsi`) installs it as
  `File /a "/oname=eiclean-backend.exe" "...\eiclean-backend-x86_64-pc-windows-msvc.exe"`
  and later `Delete "$INSTDIR\eiclean-backend.exe"` on uninstall.
- The WiX/MSI package (`target/release/wix/x64/main.wxs`) references the same unsuffixed
  `eiclean-backend.exe`.

So in both the raw build output and the installed layout, the sidecar always ends up as
`eiclean-backend.exe` in the same directory as the main app executable — no target-triple
suffix, no `tauri-plugin-shell` dependency needed to resolve it. `sidecar.rs`'s
`release_backend_exe()` relies on exactly this.

End-to-end validation: after `cargo tauri build`, launching `target/release/app.exe` directly
spawned `target/release/eiclean-backend.exe` (confirmed via `Get-Process` and a fresh
timestamped entry in `backend.log`), proving the release path resolution works without a
Python interpreter present on the machine running the installer.

## Known issues

- **Executable size (~44MB)**: `pymobiledevice3` pulls in a large transitive dependency
  footprint (xonsh, IPython, and other CLI/shell tooling it ships but this project never
  imports) that PyInstaller's static analysis can't easily prune. Functionally harmless,
  but worth revisiting (e.g. explicit `excludes` in the `.spec`) if installer size becomes a
  concern.
- **No real-device validation on this machine**: this dev environment lacks Apple Mobile
  Device Support entirely, so `device.connect`'s success path and the full pairing UX have
  only been unit/mock-tested, not exercised against a frozen exe + real iPhone. Flagged as a
  follow-up, not silently assumed to work.
- **Driver bundling/prompting** (whether the installer should bundle or prompt for Apple
  Mobile Device Support) remains an open question — see `docs/project_specification.md` §9.
